## 왜 대규모 코드베이스는 다른가?

파일이 수백 개, 줄 수가 수십만 줄인 프로젝트에서는 Claude Code를 사용하는 방식 자체가
달라져야 한다. 핵심 문제는 두 가지다.

1. **컨텍스트 한계**: 모든 코드를 한 번에 읽을 수 없다.
2. **속도와 비용**: 불필요한 컨텍스트가 많을수록 느리고 비싸진다.

이 레벨에서는 대규모 코드베이스에서 Claude Code를 **빠르고 정확하게** 활용하는 전략을 배운다.

## CLAUDE.md 최적화

대규모 프로젝트일수록 CLAUDE.md가 더 중요하다. 하지만 **길이를 늘리는 것이 아니라 간결하게 유지**하는 것이 핵심이다.

### 나쁜 예: 장황한 CLAUDE.md

```markdown
# 프로젝트 설명

이 프로젝트는 2020년에 시작된 e-commerce 플랫폼으로...
(300줄 이어서 계속)
```

### 좋은 예: 간결하고 구조적인 CLAUDE.md

```markdown
# 프로젝트: ShopCore

## 핵심 규칙
- 타입스크립트 strict 모드
- 컴포넌트: packages/ui/src/
- API: packages/api/src/routes/
- 테스트: 각 패키지 __tests__/

## 금지 사항
- any 타입 사용 금지
- console.log 커밋 금지

## 패키지 의존 규칙
ui → shared-types (단방향)
api → shared-types, db-models (단방향)

## 상세 규칙 → .claude/rules/ 참조
```

### 디렉토리 구조 가이드 포함

```markdown
## 디렉토리 구조 (핵심만)

packages/
├── ui/          # React 컴포넌트 라이브러리
├── api/         # Express API 서버
├── db-models/   # Prisma 스키마 + 모델
├── shared-types/ # 공유 TypeScript 타입
└── cli/         # 관리자 CLI 도구

scripts/         # 빌드/배포 스크립트
docs/            # 설계 문서 (코드와 분리)
```

> 구조를 먼저 파악하면 Claude가 관련 파일을 훨씬 빠르게 찾는다.

## 1M 컨텍스트 윈도우 활용 전략

Claude는 최대 약 1M 토큰의 컨텍스트를 처리할 수 있다. 이를 효과적으로 활용하는 전략이다.

### 컨텍스트 구성 우선순위

```
1순위: 작업 관련 핵심 파일 (반드시 필요)
2순위: 타입 정의, 인터페이스 (타입 안전성)
3순위: 관련 테스트 파일 (기대 동작 이해)
4순위: 설계 문서 (의도 파악)
5순위: 그 외 파일 (최대한 제외)
```

### 전략 1: 좁은 범위로 시작

```bash
# 나쁜 예: 전체 프로젝트를 한 번에
claude -p "전체 프로젝트를 분석해줘"

# 좋은 예: 목표를 명확히 하고 좁게 시작
claude --tools "Read,Glob,Grep" \
  -p "packages/api/src/routes/orders.ts 파일만 분석해서
      성능 병목이 될 수 있는 부분을 찾아줘"
```

### 전략 2: 계층적 탐색

```
1단계: 전체 구조 파악 (CLAUDE.md, 디렉토리 트리)
      ↓
2단계: 관련 패키지 식별 (package.json, index 파일)
      ↓
3단계: 핵심 파일 집중 분석 (실제 수정 대상)
```

### 전략 3: 컨텍스트를 코드로 전달

```bash
# 관련 타입만 추출해서 전달
grep -r "interface Order" packages/shared-types/src/ > /tmp/order-types.ts

claude --tools "Read" \
  -p "다음 타입 정의를 참고해서 OrderService를 구현해줘. 타입 파일: /tmp/order-types.ts"
```

## `/compact` 전략적 활용

`/compact`는 세션의 긴 히스토리를 요약해서 컨텍스트를 절약한다.

### 언제 `/compact`를 써야 하나?

```
/compact를 써야 할 시점:
- /cost 또는 /context 로 확인 시 사용량이 70% 초과
- 새로운 주제로 전환할 때
- 긴 탐색 후 실제 구현 단계로 넘어갈 때

/compact를 쓰지 말아야 할 시점:
- 작업 중간 (중요 컨텍스트가 손실될 수 있음)
- 복잡한 상태를 유지해야 할 때
```

### `/compact` 워크플로

```
1. 탐색 단계: 여러 파일 읽고 구조 파악
      ↓
2. /compact 실행: 탐색 결과를 요약으로 압축
      ↓
3. 구현 단계: 핵심 컨텍스트만 유지한 채 작업
      ↓
4. /cost 확인: 남은 컨텍스트 여유 확인
```

### `/compact` 커스텀 지시사항

```
/compact 다음 정보는 반드시 유지해줘:
- OrderService의 현재 인터페이스 시그니처
- 발견한 3가지 버그 목록
- 수정하기로 결정한 접근법
```

## Worktree를 이용한 병렬 작업

Git Worktree는 여러 기능을 **동시에 독립적으로** 개발할 수 있게 한다.

### 병렬 Worktree 설정

```bash
# 터미널 1: 기능 A 개발
claude --worktree feature-payment -p "결제 모듈 구현"

# 터미널 2: 기능 B 개발 (동시에)
claude --worktree feature-search -p "검색 기능 구현"

# 터미널 3: 버그 수정 (동시에)
claude --worktree bugfix-cart -p "장바구니 버그 수정"
```

### Worktree 병렬 작업 가이드

```
적합한 경우:
- 독립된 기능 (의존 없음)
- 서로 다른 패키지 수정
- 버그 픽스 + 기능 개발 동시 진행

적합하지 않은 경우:
- 같은 파일을 수정하는 두 작업
- 한 작업의 결과가 다른 작업의 입력인 경우
```

### tmux + Worktree 조합

```bash
# tmux 세션 생성
tmux new-session -d -s dev

# 패널 분할 후 각각 worktree 실행
tmux split-window -h
tmux send-keys -t dev:0.0 "claude --worktree feature-a" Enter
tmux send-keys -t dev:0.1 "claude --worktree feature-b" Enter
tmux attach -t dev
```

## Agent Teams로 대규모 작업 분할

단일 Claude 세션으로 처리하기 어려운 대규모 리팩토링은 **Agent Teams**로 분할한다.

```bash
# Agent Teams 활성화
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude
```

### 분할 전략 예시: 마이그레이션 작업

```
전체 작업: JavaScript → TypeScript 전환 (50개 파일)

팀원 1: packages/ui 담당 (20개 파일)
팀원 2: packages/api 담당 (15개 파일)
팀원 3: packages/shared 담당 (15개 파일)

오케스트레이터: 결과 통합 및 타입 충돌 해결
```

### 팀 작업 지시 패턴

```
오케스트레이터 프롬프트:
"다음 마이그레이션을 3개 팀으로 나눠서 병렬 진행해줘:
- 팀 1: packages/ui/src/*.js → TypeScript 전환
- 팀 2: packages/api/src/*.js → TypeScript 전환
- 팀 3: 타입 정의 shared-types 에 추가
각 팀은 독립적으로 작업하고, 완료 후 타입 충돌을 검토해줘."
```

## `.claude/rules/` 모듈별 규칙 최적화

대규모 프로젝트는 하나의 긴 CLAUDE.md 대신 **모듈별 규칙 파일**로 분리한다.

### 모노레포 규칙 파일 구조

```
.claude/rules/
├── 00-global.md        # 전체 공통 규칙
├── 01-typescript.md    # TypeScript 규칙
├── 02-testing.md       # 테스트 규칙
├── 03-api.md           # API 패키지 규칙
├── 04-ui.md            # UI 패키지 규칙
└── 05-database.md      # DB 패키지 규칙
```

### 예시: `03-api.md`

```markdown
# API 패키지 규칙

적용 범위: packages/api/

## 라우터 규칙
- 라우터 파일: src/routes/{resource}.ts
- 미들웨어: src/middleware/ 에 분리
- 에러 응답: ApiError 클래스 사용 (src/errors.ts)

## 검증 규칙
- 모든 요청은 Zod 스키마로 검증
- 스키마 파일: src/schemas/{resource}.schema.ts

## 금지 사항
- req.body 직접 접근 (검증 없이)
- 컨트롤러 내 DB 직접 쿼리
```

### 규칙 파일이 컨텍스트를 절약하는 이유

```
CLAUDE.md (항상 로드)
  → 핵심 규칙만 (100줄 이하)

.claude/rules/*.md (필요 시 로드)
  → 작업 범위에 맞는 규칙만 선택적으로 로드
  → 불필요한 규칙은 컨텍스트에서 제외
```

## 성능 최적화 팁

### `effortLevel` 설정

작업의 복잡도에 맞는 노력 수준을 지정하면 속도와 비용을 조절할 수 있다.

```bash
# 빠른 반복 작업: low
claude --effort low -p "변수명 오타 수정해줘"

# 일반 구현: (기본값 = medium)
claude -p "UserService 구현해줘"

# 복잡한 설계: high
claude --effort high -p "결제 시스템 아키텍처 설계해줘"
```

| effortLevel | 특징 | 적합한 작업 |
|-------------|------|------------|
| `low` | 빠름, 간결한 응답 | 포매팅, 단순 수정 |
| `medium` (기본) | 균형 잡힌 응답 | 일반 구현 |
| `high` | 깊은 분석, 긴 응답 | 설계, 복잡한 리팩토링 |

### Fast Mode

`/fast` 슬래시 커맨드 또는 `Alt+P` 단축키로 빠른 모드를 토글한다.

```
/fast 활성화 시:
- 같은 Opus 모델이지만 응답 속도 향상
- 탐색적 질문, 간단한 작업에 적합

/fast 비활성화:
- 표준 품질의 응답
- 중요한 구현, 설계 작업에 적합
```

### 적절한 모델 선택 전략

```
대규모 코드베이스 작업별 권장 모델:

전체 아키텍처 설계     → claude-opus-4-5 (high effort)
기능 구현             → claude-sonnet-4-5 (medium effort)
단순 수정, 리네이밍    → claude-haiku-4-5 (low effort)
보안 감사             → claude-opus-4-5 (high effort)
테스트 작성           → claude-sonnet-4-5 (medium effort)
문서화               → claude-haiku-4-5 (low effort)
```

### `additionalDirectories` 활용

```json
// .claude/settings.json
{
  "additionalDirectories": [
    "../shared-types/src",
    "../design-system/src"
  ],
  "model": "claude-sonnet-4-5",
  "effortLevel": "medium"
}
```

> 현재 패키지에서 작업 시 공통 타입과 디자인 시스템을 항상 참조할 수 있다.

## 대규모 코드베이스 탐색 패턴

### 패턴 1: Top-Down 탐색

```bash
# 1. 전체 구조 파악
claude --tools "Glob" \
  -p "프로젝트 전체 디렉토리 구조를 tree 형태로 보여줘 (깊이 2단계)"

# 2. 핵심 진입점 파악
claude --tools "Read" \
  -p "package.json, src/index.ts, src/app.ts를 읽고 이 프로젝트의 핵심 흐름을 설명해줘"

# 3. 특정 영역 깊이 분석
claude --tools "Read,Grep" \
  -p "src/services/ 폴더의 서비스들이 어떻게 연결되는지 분석해줘"
```

### 패턴 2: 버그 추적 (Bottom-Up)

```bash
# 에러 메시지에서 시작
claude --tools "Grep,Read" \
  -p "
    에러 메시지: 'Cannot read property userId of undefined'
    이 에러가 발생할 수 있는 모든 코드 경로를 추적해줘.
    src/ 디렉토리에서 검색을 시작해줘.
  "
```

### 패턴 3: 영향도 분석

```bash
# 수정 전 영향도 파악
claude --tools "Grep,Read" \
  -p "
    UserService의 getUserById 메서드를 수정하려고 해.
    이 메서드를 호출하는 모든 파일과 영향을 받는 컴포넌트를 찾아줘.
  "
```

## 실습 예시

### 실습 1: 모노레포 CLAUDE.md 세팅

```bash
# 프로젝트 루트
cat > CLAUDE.md << 'EOF'
# ShopCore 모노레포

## 패키지 구조
- packages/ui: React 컴포넌트 (Storybook)
- packages/api: Express REST API
- packages/db: Prisma ORM
- packages/shared: 공유 타입, 유틸

## 빌드
npm run build        # 전체 빌드
npm run build --filter=ui   # 특정 패키지

## 규칙
.claude/rules/ 참조
EOF

# 모듈별 규칙 파일 생성
mkdir -p .claude/rules
echo "# UI 규칙 - 컴포넌트는 Storybook 스토리 필수" > .claude/rules/ui.md
echo "# API 규칙 - 라우터에서 비즈니스 로직 분리" > .claude/rules/api.md
```

### 실습 2: 대규모 리팩토링 병렬 처리

```bash
# 터미널 분할 후 동시 실행

# 터미널 1: API 패키지 리팩토링
claude --worktree refactor-api \
  --add-dir packages/shared/src \
  --tools "Edit,Read,Bash(npm run type-check)" \
  -p "packages/api/src 의 콜백 기반 코드를 async/await로 전환해줘"

# 터미널 2: UI 패키지 리팩토링 (동시에)
claude --worktree refactor-ui \
  --add-dir packages/shared/src \
  --tools "Edit,Read,Bash(npm run build --filter=ui)" \
  -p "packages/ui/src 의 클래스 컴포넌트를 함수형으로 전환해줘"
```

### 실습 3: 컨텍스트 효율적인 버그 수정

```bash
# 1단계: 영향 범위 파악 (읽기 전용)
claude --tools "Grep,Read" --effort low \
  -p "OrderController에서 발생하는 null 참조 에러 원인 찾아줘"

# 2단계: /compact 후 수정
# (탐색 결과를 압축)
# /compact 핵심 정보: null 원인은 line 47, userId 검증 누락

# 3단계: 실제 수정 (Edit만 허용)
claude --tools "Edit,Bash(npm test)" --effort medium \
  -p "방금 찾은 null 참조 문제를 수정하고 테스트 추가해줘"
```

## 성능 체크리스트

```
대규모 프로젝트 세션 시작 전:
□ CLAUDE.md가 100줄 이하인가?
□ .claude/rules/ 에 모듈별 규칙이 분리되어 있는가?
□ additionalDirectories에 꼭 필요한 것만 있는가?
□ 작업 범위가 명확히 좁혀져 있는가?

작업 중:
□ /context 또는 /cost 로 컨텍스트 사용량 확인
□ 70% 초과 시 /compact 또는 새 세션 고려
□ 독립 작업은 Worktree로 병렬화

모델 선택:
□ 설계/아키텍처 → Opus
□ 일반 구현 → Sonnet
□ 단순 수정/문서화 → Haiku
```

## Scaling Claude: 조직 확장 전략

팀이나 조직 전체에서 Claude Code를 표준화하고 확장하는 전략이다.

### 조직 단위 설정 배포

```json
// 관리 설정 파일 (managed-settings.json) — IT 팀이 배포
// 최고 우선순위로, 개인이 재정의할 수 없음
{
  "model": "sonnet",
  "permissions": {
    "deny": ["Bash(rm -rf *)", "Bash(curl *)"],
    "allow": ["Bash(npm *)"]
  }
}
```

### 팀 온보딩 표준화

```
1. 저장소에 CLAUDE.md + .claude/rules/ 포함
     → 팀원이 clone하면 자동으로 규칙 적용

2. .mcp.json으로 MCP 서버 통일
     → 팀 전체가 같은 외부 도구 사용

3. .claude/settings.json으로 권한 통일
     → 위험한 명령어를 팀 레벨에서 차단

4. .claude/agents/로 표준 에이전트 공유
     → 코드 리뷰, 테스트, 배포 에이전트 통일

5. .claude/skills/로 워크플로 표준화
     → 배포, PR 생성 등 스킬을 팀 전체가 공유
```

### CI/CD 파이프라인 통합

```yaml
# GitHub Actions에서 Claude Code 활용
# .github/workflows/claude-review.yml 예시
name: Claude Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: Run Code Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          # 변경된 파일만 리뷰
          claude -p "이 PR의 변경사항을 리뷰하고 보안, 성능, 코드 품질 이슈를 찾아줘" \
            --tools "Read,Glob,Grep" \
            --output-format json \
            --max-budget-usd 2.00 \
            --max-turns 10
```

### 멀티 프로젝트 관리

```bash
# 여러 프로젝트에서 일관된 Claude 환경 유지
~/.claude/CLAUDE.md           # 조직 공통 규칙
~/.claude/settings.json       # 공통 설정
~/.claude/skills/deploy/      # 공통 배포 스킬

# 프로젝트별 커스터마이징
project-a/CLAUDE.md           # A 프로젝트 고유 규칙
project-b/CLAUDE.md           # B 프로젝트 고유 규칙
```

### 사용량 모니터링 및 거버넌스

```bash
# /cost로 팀원별 사용량 추적
claude -p "비용 분석" --output-format json | jq '.cost_usd'

# --max-budget-usd로 팀원별 예산 제한
claude --max-budget-usd 10.00

# 환경변수로 조직 정보 활용
echo $CLAUDE_CODE_ORGANIZATION_UUID  # 소속 조직
echo $CLAUDE_CODE_USER_EMAIL         # 사용자 이메일
echo $CLAUDE_CODE_ACCOUNT_UUID       # 계정 UUID
```

### 확장 시 체크리스트

```
□ CLAUDE.md가 저장소에 포함되어 있는가?
□ .claude/rules/에 팀 코딩 규칙이 있는가?
□ .claude/settings.json으로 위험 명령이 차단되어 있는가?
□ .mcp.json으로 필수 MCP 서버가 설정되어 있는가?
□ managed-settings.json이 IT 팀에 의해 배포되었는가?
□ CI/CD에서 --max-budget-usd가 설정되어 있는가?
□ 팀 온보딩 문서에 Claude Code 설정 가이드가 있는가?
```

## 핵심 정리

```
CLAUDE.md 최적화    → 간결하게, 구조 가이드 포함
1M 컨텍스트 전략   → 좁은 범위 + 계층적 탐색
/compact            → 70% 초과 시, 단계 전환 시 활용
Worktree 병렬화    → 독립 작업을 동시에
Agent Teams        → 대규모 분할 작업
.claude/rules/     → 모듈별 규칙 분리로 컨텍스트 절약
effortLevel        → 작업 복잡도에 맞게 조절
모델 선택          → 설계=Opus, 구현=Sonnet, 단순=Haiku
Scaling Claude     → 조직 설정 배포, CI/CD 통합, 사용량 거버넌스
```
