## Claude Code 가격 이해

효율적으로 사용하려면 먼저 **비용이 어떻게 발생하는지** 알아야 한다.
Claude Code는 대화할 때마다 토큰을 소비하며, 이것이 비용의 핵심이다.

### 구독 플랜별 차이

```
플랜별 Claude Code 사용 한도:

Pro ($20/월)
  - Claude Code 포함
  - 월별 사용량 한도 있음
  - 한도 초과 시 추가 비용 발생

Max ($100/월 또는 $200/월)
  - 더 높은 사용량 한도
  - $100: Pro의 5배 한도
  - $200: Pro의 20배 한도

Team (사용자당 $25~30/월)
  - 팀원별 독립 한도
  - 관리 콘솔 제공

Enterprise (협의)
  - 무제한 또는 커스텀 한도
  - 전용 용량, SLA 보장
```

### 토큰 사용량과 비용 관계

```
토큰(Token)이란?
  → 텍스트를 처리하는 단위 (대략 단어의 3/4 크기)
  → 영어 1단어 ≈ 1.3 토큰
  → 한국어 1글자 ≈ 2~3 토큰

비용이 발생하는 토큰:
  입력 토큰: Claude에게 보내는 모든 것
    - 대화 히스토리 (누적됨!)
    - 읽는 파일 내용
    - CLAUDE.md 내용
    - 시스템 프롬프트

  출력 토큰: Claude가 생성하는 응답
    - 텍스트 응답
    - 생성된 코드

  캐시 토큰: 반복 사용 시 할인 적용
    - 동일 컨텍스트 재사용 시 비용 절감
```

### `/cost`로 확인하기

```bash
# 현재 세션의 토큰 사용량과 비용 확인
/cost

# 출력 예시:
# 입력 토큰: 45,230
# 출력 토큰: 8,120
# 캐시 조회: 32,100
# 예상 비용: $0.87
```

> `/cost`는 세션 중 언제든 입력할 수 있다.
> 비용이 예상보다 빠르게 증가한다면 전략을 바꿀 신호다.

## 토큰 사용량 줄이기

### 컨텍스트에 필요한 것만 포함

Claude가 파일을 읽을 때마다 해당 파일 전체가 컨텍스트(입력 토큰)에 들어간다.
불필요한 파일을 읽히지 않는 것만으로도 비용을 크게 줄일 수 있다.

```bash
# 비효율적: 전체 소스 디렉토리를 읽힘
claude --tools "Read" \
  -p "src/ 전체를 분석해서 버그를 찾아줘"

# 효율적: 필요한 파일만 지정
claude --tools "Read,Grep" \
  -p "결제 로직 버그를 찾아줘.
      먼저 Grep으로 'payment' 관련 파일을 찾고
      핵심 파일만 읽어줘"
```

```bash
# 비효율적: 전체 파일을 읽고 일부만 사용
claude -p "@src/bigFile.ts 에서 formatDate 함수만 수정해줘"
# → bigFile.ts 전체가 토큰을 소비함

# 효율적: 필요한 함수 위치를 먼저 파악
claude --tools "Grep" \
  -p "formatDate 함수가 있는 정확한 파일과 라인을 찾아줘"
# → 이후 해당 부분만 수정 요청
```

### 불필요한 파일 읽기 방지

```json
// .claude/settings.json - 자동 읽기를 제한
{
  "autoRead": false,
  "tools": {
    "allowedPaths": [
      "src/**",
      "tests/**"
    ],
    "blockedPaths": [
      "node_modules/**",
      "dist/**",
      "*.lock",
      "coverage/**"
    ]
  }
}
```

```markdown
<!-- CLAUDE.md - Claude에게 읽기 범위 안내 -->
## 파일 읽기 규칙

다음 파일/폴더는 읽지 않아도 됨 (자동 생성 파일):
- node_modules/, dist/, .next/, coverage/
- *.lock 파일 (package-lock.json, yarn.lock)
- *.map 파일

작업 전 반드시 먼저 Grep으로 관련 파일을 찾고,
필요한 파일만 Read할 것.
```

### `effortLevel` 조절

작업의 복잡도에 맞는 노력 수준을 설정하면 비용을 최적화할 수 있다.

```bash
# 단순 작업: low (빠르고 저렴)
claude --effort low -p "이 변수명을 camelCase로 바꿔줘"

# 일반 작업: medium (기본값, 균형)
claude -p "이 서비스 클래스를 구현해줘"

# 복잡한 작업: high (느리지만 정확)
claude --effort high -p "이 시스템의 보안 취약점을 깊이 분석해줘"
```

| effortLevel | 토큰 소비 | 적합한 작업 |
|-------------|----------|------------|
| `low` | 낮음 | 포매팅, 오타 수정, 단순 질문 |
| `medium` (기본) | 중간 | 기능 구현, 일반 버그 수정 |
| `high` | 높음 | 아키텍처 설계, 보안 감사, 복잡한 리팩토링 |

### 적절한 모델 선택

비싼 모델이 항상 좋은 것은 아니다. 작업에 맞는 모델을 쓰면 비용 대비 결과가 더 좋다.

```bash
# 단순 작업 → Haiku (가장 저렴, 빠름)
claude --model claude-haiku-4-5 \
  -p "이 함수에 JSDoc 주석을 달아줘"

# 일반 구현 → Sonnet (가성비 최고)
claude --model claude-sonnet-4-5 \
  -p "UserProfile 컴포넌트를 구현해줘"

# 복잡한 설계 → Opus (가장 비싸지만 정확)
claude --model claude-opus-4-5 \
  -p "마이크로서비스 아키텍처를 설계해줘"
```

```
작업별 권장 모델:

단순 수정, 문서화, 포매팅  → claude-haiku-4-5
기능 구현, 버그 수정       → claude-sonnet-4-5
아키텍처, 보안, 딥 분석    → claude-opus-4-5
```

## 컨텍스트 윈도우 관리

컨텍스트 윈도우는 Claude가 한 번에 처리할 수 있는 최대 정보량이다.
세션이 길어질수록 이 공간이 채워지고, 비용도 증가한다.

### `/context`로 사용량 모니터링

```bash
# 현재 컨텍스트 사용량 확인
/context

# 출력 예시:
# 컨텍스트 사용량: 68% (680K / 1M 토큰)
# 경고: 80% 초과 시 성능 저하 가능
```

### `/compact` 시기와 방법

`/compact`는 긴 대화 히스토리를 요약하여 컨텍스트를 절약한다.

```bash
# 컨텍스트가 70% 이상 차면 /compact 실행
/compact

# 중요한 정보를 유지하도록 지시도 가능
/compact 다음 정보는 반드시 유지해줘:
- 발견한 버그 목록 (3개)
- 수정하기로 결정한 접근법
- 현재 작업 중인 파일: src/services/authService.ts
```

```
/compact 사용 타이밍:

써야 할 때:
  - 컨텍스트가 70% 초과
  - 새로운 주제로 전환할 때
  - 긴 탐색을 마치고 구현 단계로 전환할 때

쓰지 말아야 할 때:
  - 복잡한 작업 중간 (중요 컨텍스트 손실 위험)
  - 여러 파일의 복잡한 의존 관계를 추적 중일 때
```

### 자동 압축 (auto-compaction) 이해

```json
// .claude/settings.json
{
  "autoCompact": true,           // 자동 압축 활성화 (기본값)
  "autoCompactThreshold": 0.8    // 80% 초과 시 자동으로 /compact 실행
}
```

> 자동 압축이 켜져 있으면 Claude가 알아서 컨텍스트를 관리한다.
> 하지만 중요한 작업 중에는 `autoCompact: false`로 끄고 수동으로 관리하는 것이 안전하다.

### 새 세션 vs 이어하기 판단 기준

```
새 세션을 시작해야 할 때:
  - 완전히 다른 주제/기능으로 전환
  - 컨텍스트가 90% 이상 차서 /compact로도 부족할 때
  - 이전 대화가 현재 작업과 무관할 때
  - 새벽에 잠자고 다음날 작업 재개할 때

이어서 작업해야 할 때:
  - 같은 기능을 계속 개발 중
  - 이전 대화의 결정사항과 맥락이 필요할 때
  - 복잡한 버그를 중간에 추적 중일 때
```

## Thinking 모드 활용

Extended Thinking은 Claude가 더 깊이 생각하는 기능이지만, 비용이 크게 증가한다.

### Extended Thinking이 비용에 미치는 영향

```
Extended Thinking (Alt+T) 켰을 때:
  - 내부 추론 과정도 토큰으로 소비
  - 일반 모드 대비 3~5배 토큰 사용
  - 응답 시간도 증가

예시:
  일반 모드: 응답 500토큰
  Thinking 모드: 추론 2000토큰 + 응답 500토큰 = 2500토큰
```

### 언제 Extended Thinking을 켤 것인가

```
Thinking 모드가 유용한 경우 (비용 대비 가치 있음):
  - 복잡한 알고리즘 설계
  - 깊은 아키텍처 트레이드오프 분석
  - 어려운 버그의 근본 원인 파악
  - 여러 선택지 사이의 복잡한 판단

Thinking 모드가 불필요한 경우 (낭비):
  - 간단한 코드 수정
  - 파일 읽기/탐색
  - 명확한 요구사항의 단순 구현
  - 포매팅, 이름 변경
```

```bash
# Thinking 모드를 특정 질문에만 한정 사용
# Alt+T 로 토글하거나 CLI에서:
claude --model claude-opus-4-5 \
  -p "think deeply: 이 분산 캐시 설계의 일관성 트레이드오프를 분석해줘"
# → "think deeply" 키워드로 Extended Thinking 유도
```

## 서브에이전트와 Hook 활용

### 서브에이전트 비용 구조 이해

```
서브에이전트(Task 도구)를 사용하면:
  - 새로운 Claude 인스턴스가 생성됨
  - 각 서브에이전트가 독립적으로 토큰 소비
  - 오케스트레이터 + 서브에이전트 토큰 합산

비용 최적화:
  - 서브에이전트에 적절한 모델 지정
  - 불필요한 서브에이전트 생성 방지
  - 독립 작업만 병렬 서브에이전트로 처리
```

```bash
# 비효율적: 모든 작업에 Opus 서브에이전트
Task(subagent_type="executor", model="opus", prompt="주석 달아줘")

# 효율적: 작업 복잡도에 맞는 모델 지정
Task(subagent_type="executor", model="haiku", prompt="주석 달아줘")
Task(subagent_type="executor", model="sonnet", prompt="기능 구현해줘")
Task(subagent_type="executor", model="opus", prompt="아키텍처 설계해줘")
```

### Hook으로 불필요한 작업 차단하여 비용 절약

Hook을 사용하면 Claude가 특정 도구를 사용하기 전에 검사할 수 있다.
불필요한 파일 읽기나 위험한 명령을 차단하여 낭비를 줄인다.

```json
// .claude/settings.json - Hook으로 비용 절약
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read",
        "hooks": [
          {
            "type": "command",
            "command": "check-file-size.sh",
            "description": "대용량 파일 읽기 차단 (1MB 이상)"
          }
        ]
      }
    ]
  }
}
```

```bash
#!/bin/bash
# check-file-size.sh - 대용량 파일 읽기 차단
# (파일이 너무 크면 불필요한 토큰 낭비)

FILE_PATH=$(echo "$CLAUDE_TOOL_INPUT" | jq -r '.path')
FILE_SIZE=$(stat -f%z "$FILE_PATH" 2>/dev/null || echo 0)

# 1MB 이상이면 경고 메시지 출력
if [ "$FILE_SIZE" -gt 1048576 ]; then
  echo "경고: $FILE_PATH 는 ${FILE_SIZE} bytes로 너무 큽니다."
  echo "필요한 부분만 Grep으로 검색하는 것을 권장합니다."
  exit 2  # Claude에게 경고 메시지 전달
fi

exit 0  # 정상 진행
```

## 실전 비용 절약 팁 10가지

```
1. 세션 시작 전 범위를 명확히 하라
   → 탐색 단계에서 읽을 파일을 미리 파악하면 불필요한 읽기를 줄일 수 있다

2. Haiku로 먼저 탐색하고, Sonnet/Opus로 구현하라
   → "어디를 수정해야 하나?" 탐색은 Haiku, 실제 구현은 Sonnet

3. /compact를 습관화하라
   → 컨텍스트 70% 초과 전에 주기적으로 /compact 실행

4. CLAUDE.md로 반복 설명을 제거하라
   → 매번 코딩 규칙을 설명하는 것도 토큰 낭비

5. --tools 로 도구를 제한하라
   → 읽기만 필요한 작업에 Edit 도구가 활성화되어 있으면 낭비

6. 큰 파일은 Grep으로 먼저 검색하라
   → 1000줄 파일 전체를 읽히는 대신 필요한 함수만 찾아서 읽기

7. node_modules, dist 를 차단하라
   → .gitignore 와 유사하게 .claude/settings.json 에서 blockedPaths 설정

8. Extended Thinking은 꼭 필요할 때만 켜라
   → 일반 질문에는 꺼두고, 복잡한 분석에만 활성화

9. /cost를 정기적으로 확인하라
   → 비용 인식이 낭비를 줄이는 첫 번째 단계

10. 완성된 기능은 새 세션에서 시작하라
    → 긴 히스토리를 계속 이어가는 것보다 새 세션이 효율적일 때가 많다
```

## 비용 모니터링 워크플로

```bash
#!/bin/bash
# cost-aware-session.sh - 비용 인식 세션 스크립트

# 1. 세션 시작 시 예산 설정 (메모용)
echo "오늘 예산: $2.00"

# 2. 작업 중 주기적으로 /cost 확인
# (Claude와 대화하다가 중간에 /cost 입력)

# 3. 세션 종료 후 사용량 기록
# (스프레드시트나 메모에 날짜/작업/비용 기록)
```

```
비용 효율성 체크리스트:

세션 시작 전:
□ 작업 범위가 명확한가?
□ 적절한 모델을 선택했는가? (Haiku/Sonnet/Opus)
□ effortLevel을 작업에 맞게 설정했는가?

작업 중:
□ /cost를 마지막으로 확인한 것이 30분 이내인가?
□ 불필요한 파일을 읽히지는 않았는가?
□ Extended Thinking이 필요 없는데 켜져 있지는 않은가?

세션 종료 전:
□ /cost로 최종 사용량 확인
□ 다음 세션을 위해 핵심 결과물을 파일로 저장
□ 새 세션 시작이 더 효율적인 상황은 아닌가?
```

## 핵심 정리

```
/cost             → 현재 세션 비용 확인
/context          → 컨텍스트 사용량 모니터링
/compact          → 70% 초과 시 컨텍스트 압축

effortLevel       → low/medium/high 로 비용 조절
모델 선택         → Haiku(탐색) > Sonnet(구현) > Opus(설계)
```
