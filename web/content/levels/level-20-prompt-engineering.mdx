## 프롬프트 엔지니어링이 왜 중요한가?

Claude Code는 강력하지만, 결국 **어떻게 말하느냐**가 결과의 질을 결정한다.
같은 작업을 요청하더라도 프롬프트를 어떻게 구성하느냐에 따라 결과물의 품질,
속도, 정확도가 크게 달라진다.

프롬프트 엔지니어링은 "AI를 속이는 마법"이 아니라, **명확하게 소통하는 기술**이다.
개발자가 동료에게 작업을 설명할 때 좋은 설명과 나쁜 설명이 있듯이, Claude에게도
마찬가지다.

## 효과적인 프롬프트 패턴

### 패턴 1: 구체적이고 명확한 지시

모호한 요청은 모호한 결과를 만든다. 목표, 범위, 제약을 명시하라.

```
# 나쁜 예: 모호한 지시
"코드 개선해줘"

# 좋은 예: 구체적인 지시
"src/services/userService.ts 파일의 getUserById 함수를
 에러 핸들링을 추가하여 개선해줘.
 - null 반환 대신 커스텀 에러(UserNotFoundError)를 throw
 - DB 연결 에러도 별도 처리
 - 기존 테스트가 깨지지 않게 유지"
```

```
# 나쁜 예: 결과 형식 불명확
"이 API 문서화해줘"

# 좋은 예: 출력 형식 명시
"routes/api/users.ts 의 엔드포인트를 JSDoc 형식으로 문서화해줘.
 각 엔드포인트에 @param, @returns, @throws, @example 태그 포함.
 기존 코드는 수정하지 말고 주석만 추가해줘."
```

### 패턴 2: 맥락 제공하기 (@파일 참조)

Claude는 참조할 파일이 없으면 추측에 의존한다. 관련 파일을 명시적으로 제공하라.

```
# 나쁜 예: 맥락 없이 요청
"새 API 엔드포인트 추가해줘"

# 좋은 예: 관련 파일과 맥락 제공
"@routes/api/users.ts 와 @schemas/user.schema.ts 를 참고해서
 POST /api/users/:id/avatar 엔드포인트를 추가해줘.
 기존 엔드포인트와 동일한 패턴으로 구현하고,
 multer 미들웨어로 이미지 업로드를 처리해줘."
```

```
# 관련 타입 정의를 함께 제공
"@types/order.ts 의 Order 인터페이스를 참고해서
 OrderSummary 컴포넌트를 만들어줘.
 기존 @components/ProductCard.tsx 와 같은 스타일로."
```

### 패턴 3: 단계적 작업 분해

한 번에 너무 많은 것을 요청하면 Claude가 방향을 잃거나 중요한 부분을 빠뜨릴 수 있다.

```
# 나쁜 예: 한 번에 모든 것을 요청
"쇼핑몰 장바구니 기능을 완전히 구현해줘.
 DB 스키마, API, 프론트엔드, 테스트까지 다 해줘."

# 좋은 예: 단계적으로 나눠서 요청
# 1단계: 설계
"장바구니 기능을 위한 DB 스키마를 설계해줘.
 User, Product, CartItem 테이블 관계도 포함해서."

# 2단계: 백엔드
"방금 설계한 스키마 기반으로 CartService를 구현해줘.
 CRUD 메서드만 우선 구현하고, 결제 연동은 나중에."

# 3단계: 프론트엔드
"CartService API를 사용하는 CartPage 컴포넌트를 구현해줘."

# 4단계: 테스트
"CartService의 단위 테스트를 작성해줘."
```

### 패턴 4: 출력 형식 지정

원하는 결과의 형태를 명확히 지정하면 Claude가 그에 맞게 구조화된 응답을 제공한다.

```
# JSON 형식으로 분석 결과 요청
"src/ 디렉토리를 분석해서 다음 JSON 형식으로 반환해줘:
{
  'topLevelModules': [...],
  'complexity': { 'high': [...], 'medium': [...] },
  'suggestions': [...]
}"

# 마크다운 테이블로 비교
"claude-opus-4-5, claude-sonnet-4-5, claude-haiku-4-5 세 모델을
 속도/비용/품질 기준으로 마크다운 테이블로 비교해줘."

# 번호 목록으로 실행 계획
"이 리팩토링 작업을 실행 순서대로 번호 목록으로 정리해줘.
 각 단계에 예상 소요 시간도 포함해줘."
```

## Claude Code 워크플로 최적화

### 탐색 → 계획 → 실행 → 검증 패턴

```
1단계: 탐색 (도구: Read, Glob, Grep)
  - 관련 파일과 코드 구조 파악
  - "이 파일들을 읽고 현재 구조를 설명해줘"
        ↓
2단계: 계획 (permissionMode: plan)
  - 변경 범위와 접근법 수립
  - Shift+Tab 으로 plan 모드 진입
  - "어떤 파일을 어떻게 수정할지 계획만 세워줘"
        ↓
3단계: 실행 (도구: Edit, Write, Bash)
  - 계획한 대로 구현
  - "계획한 대로 구현해줘"
        ↓
4단계: 검증 (도구: Bash)
  - 테스트와 타입 체크로 결과 확인
  - "npm test && npm run type-check 실행해줘"
```

### `/plan` 모드를 활용한 계획적 접근

큰 작업을 시작하기 전에 계획 단계를 거치면 불필요한 수정을 줄일 수 있다.

```
# Shift+Tab 으로 plan 모드 진입 후:
"인증 시스템을 JWT에서 세션 기반으로 마이그레이션하는
 계획을 세워줘. 수정 필요한 파일 목록과 순서를 포함해줘."

# Claude가 계획을 제시하면:
# - 계획 검토 후 조정 요청
# - 동의하면 plan 모드 해제 후 실행 단계로
```

### 반복적 개선 접근법

처음부터 완벽할 필요가 없다. 빠르게 동작하는 버전을 만들고 개선하는 것이 효과적이다.

```
# 1차: 기능 우선, 간단하게
"UserCard 컴포넌트를 일단 동작하게만 만들어줘.
 스타일은 나중에 다듬을게."

# 2차: 에러 처리 추가
"UserCard에 로딩 상태와 에러 상태를 추가해줘."

# 3차: 성능 최적화
"UserCard에 React.memo와 useMemo를 추가해서 최적화해줘."

# 4차: 접근성 개선
"UserCard에 ARIA 속성과 키보드 네비게이션을 추가해줘."
```

## 안티패턴: 피해야 할 것들

### 안티패턴 1: 너무 모호한 지시

```
# 피해야 할 패턴
"더 좋게 만들어줘"       # '더 좋게'의 기준이 없음
"버그 고쳐줘"           # 어떤 버그인지 불명확
"리팩토링해줘"          # 방향이 없음
"최신 패턴으로 바꿔줘"  # '최신'의 기준이 없음

# 대신 이렇게
"로그인 실패 시 에러 메시지가 표시되지 않는 버그를 고쳐줘.
 재현 방법: 잘못된 비밀번호 입력 후 확인 버튼 클릭"
```

### 안티패턴 2: 한 번에 너무 많은 것을 요청

```
# 피해야 할 패턴 (한 번에 5가지 이상)
"이 파일에서 버그 고치고, 타입 추가하고, 테스트 작성하고,
 문서화하고, 성능 최적화하고, 스타일도 개선해줘."

# 대신 이렇게 (작업을 순서대로 나눠서)
# 1번: "이 파일의 버그를 먼저 고쳐줘"
# 2번: "이제 타입 정의를 추가해줘"
# 3번: "방금 수정한 함수에 대한 테스트를 작성해줘"
```

### 안티패턴 3: 맥락 없이 수정 요청

```
# 피해야 할 패턴
"아까 만든 함수 수정해줘"   # 어떤 함수인지 불명확
"이전 방식으로 되돌려줘"    # 어떤 변경을 되돌리는지 불명확

# 대신 이렇게
"src/utils/format.ts 의 formatDate 함수를
 ISO 8601 대신 'YYYY년 MM월 DD일' 형식으로 수정해줘."
```

### 안티패턴 4: 과도한 기대

```
# 현실적이지 않은 요청
"전체 앱을 한 번에 만들어줘"
"모든 엣지케이스를 다 처리해줘"
"완벽한 코드를 써줘"

# 현실적인 접근
"주요 흐름 먼저 구현 → 에러 처리 추가 → 엣지케이스 대응"
순서로 단계적으로 요청하는 것이 효과적이다.
```

## CLAUDE.md를 활용한 일관성 확보

매번 같은 규칙을 반복해서 말하는 것은 낭비다. CLAUDE.md에 고정해두면 자동으로 적용된다.

### 반복 지시 대신 CLAUDE.md에 규칙 고정

```markdown
# 프로젝트 코딩 규칙 (CLAUDE.md)

## 언어 및 스타일
- 모든 코드 주석은 한국어로 작성
- 변수명, 함수명은 camelCase (영어)
- 들여쓰기: 2칸 스페이스

## 에러 처리
- async 함수는 항상 try/catch 포함
- 사용자에게 표시되는 에러 메시지는 한국어로
- 로깅: console.error 대신 logger.error() 사용

## 컴포넌트 규칙 (React)
- 함수형 컴포넌트만 사용
- Props 타입은 별도 interface로 정의
- 컴포넌트 파일명: PascalCase.tsx
```

### 프로젝트별 코딩 컨벤션 설정

```markdown
# API 개발 규칙

## 엔드포인트 네이밍
- 복수형 리소스명: /api/users, /api/products
- 동사 사용 금지: /api/getUser (X) → /api/users/:id (O)

## 응답 형식 (항상 이 구조 사용)
{
  "success": boolean,
  "data": any | null,
  "error": string | null,
  "meta": { "timestamp": string }
}

## 인증
- 모든 보호 엔드포인트에 authMiddleware 적용
- JWT 토큰은 Authorization: Bearer 헤더로
```

> CLAUDE.md는 "매번 설명하기 귀찮은 것들"을 저장하는 곳이다.
> 한 번 잘 써두면 이후 모든 세션에서 자동으로 적용된다.

## 실습 예시: 좋은 프롬프트 vs 나쁜 프롬프트

### 실습 세트 1: 코드 리뷰

```
# 나쁜 프롬프트
"코드 리뷰해줘"

# 좋은 프롬프트
"@src/services/paymentService.ts 를 코드 리뷰해줘.
 다음 기준으로 평가해줘:
 1. 보안 취약점 (SQL Injection, XSS 등)
 2. 에러 처리 누락
 3. 성능 문제 (N+1 쿼리 등)
 심각도를 [높음/중간/낮음]으로 분류하고,
 각 이슈에 수정 코드 예시를 포함해줘."
```

### 실습 세트 2: 새 기능 구현

```
# 나쁜 프롬프트
"검색 기능 추가해줘"

# 좋은 프롬프트
"@components/ProductList.tsx 에 검색 기능을 추가해줘.

 요구사항:
 - 상품명과 설명 기준으로 클라이언트 사이드 필터링
 - 디바운스 300ms 적용 (불필요한 렌더링 방지)
 - 검색어 없을 때는 전체 목록 표시
 - 검색 결과 없을 때 '검색 결과가 없습니다' 메시지

 @types/product.ts 의 Product 타입 참고해서
 기존 코드 스타일과 일관성 유지해줘."
```

### 실습 세트 3: 버그 수정

```
# 나쁜 프롬프트
"로그인이 안 돼. 고쳐줘"

# 좋은 프롬프트
"로그인 기능 버그를 수정해줘.

 증상:
 - 올바른 아이디/비밀번호 입력해도 '인증 실패' 에러 발생
 - 브라우저 콘솔: 'TypeError: Cannot read property id of undefined'

 재현 방법:
 1. /login 페이지 접속
 2. 등록된 이메일과 비밀번호 입력
 3. 로그인 버튼 클릭

 관련 파일:
 @pages/LoginPage.tsx
 @services/authService.ts
 @api/auth.ts

 최근 변경: 어제 authService.ts 에서 토큰 처리 로직 수정함"
```

### 실습 세트 4: 리팩토링

```
# 나쁜 프롬프트
"이 코드 리팩토링해줘"

# 좋은 프롬프트
"@utils/dateUtils.ts 를 다음 방향으로 리팩토링해줘:

 목표:
 - 중복 코드 제거 (formatDate, formatDateTime이 유사한 로직)
 - date-fns 라이브러리 사용 (현재 직접 구현된 부분 교체)
 - TypeScript 타입 강화 (string 대신 구체적인 타입)

 제약:
 - 기존 함수명과 시그니처는 유지 (하위 호환성)
 - 기존 테스트가 모두 통과해야 함
 - 새 의존성 추가 없이 이미 설치된 date-fns만 사용"
```

## 핵심 정리

```
구체적인 지시     → 목표, 범위, 제약을 명시하라
맥락 제공        → @파일명으로 관련 파일을 참조하라
단계적 분해      → 큰 작업을 작은 단계로 나눠서 요청하라
출력 형식 지정   → JSON, 테이블, 번호 목록 등 형식을 명시하라
계획 먼저       → /plan 모드로 실행 전 방향을 수립하라
반복적 개선     → 동작하는 버전 먼저, 개선은 단계적으로
CLAUDE.md 활용  → 반복 규칙은 파일에 고정, 매번 설명하지 말 것
안티패턴 피하기 → 모호함, 과다 요청, 맥락 없는 수정 금지
```
